# 项目主要目标

## 1. 项目定位

核心思想是让用户通过一句话或数轮简单对话，触发一个“受约束的自动生成流程”，最终完成硬件方案生成、逻辑脚本生成、部署和运行。

这个项目不应被定义为“AI 任意写代码并直接控制硬件”，而应被定义为：

1. 用户用自然语言描述需求。
2. AI 负责理解需求、拆解流程、选择可用模组。
3. 平台负责校验、生成、签名、发布、回滚。
4. 主机负责运行、保护和监控。

目标是：

1. 最大化简化操作，让 0 基础用户可以用自然语言完成简单项目。
2. 将复杂的硬件接线、协议调用、代码模板和调试过程隐藏在系统内部。
3. 让 AI 负责“理解需求和组装方案”，而不是直接绕过规则操作硬件。
4. 在保证体验简单的同时，保留工程上的安全性、可验证性和可回滚能力。

## 2. 面向对象

1. 孩童和初学者。
2. 0 基础但希望快速验证创意的小白用户。
3. 需要快速出样的个人开发者、教育场景使用者、简单项目制作者。

## 3. 短期内非目标

首期不追求以下目标：

1. 不追求支持全部传感器和执行器。
2. 不追求让 AI 自由生成完整固件。
3. 不追求复杂多任务实时控制。
4. 不追求工业级高风险控制场景。
5. 不追求第一期就做完整社区自动审核体系。

## 4. 核心原则

1. AI 只负责生成“结构化方案”和“受约束逻辑脚本”，不能绕过平台规则直接控制硬件。
2. 系统必须先知道“用户手上实际有哪些模组、这些模组支持什么能力”，再允许 AI 生成方案。
3. 所有下位机能力都通过统一协议和统一能力描述暴露给主机。
4. 所有部署必须经过静态校验、沙盒验证和签名发布。
5. 一期先做少量高频、低风险模组，先跑通闭环，再扩展模组生态。
6. 真正的安全边界由白名单能力、校验器和主机运行时保护共同组成。

## 5. 关键技术

1. 主机侧 LuatOS 沙盒运行环境，可实现脚本热注入、隔离执行、异常退出保护和快速回滚。
2. 设备能力发现机制，用于上报模组种类、固件版本、支持操作、参数范围和资源占用。
3. 结构化任务 DSL，用于把自然语言需求转换为可验证的中间表示。
4. 受约束代码生成器，用模板和能力映射生成逻辑脚本，而不是让模型自由输出全部运行逻辑。
5. CANFD 私有协议 v1，用于统一主机与模组之间的数据交换、查询、控制、事件上报和 OTA。
6. 云端发布闸门，包括静态检查、依赖检查、资源预算检查、危险动作限制、沙盒测试和签名下发。
7. 主机运行时保护，包括 `send_cmd` 白名单验证、频率限制、超时断开、健康检查和自动回滚。

# 术语定义

为避免后续讨论混乱，统一使用以下术语：

1. `主机`：用户手上的核心设备，负责联网、录音、显示、拉取版本、运行逻辑脚本和转发协议。
2. `模组`：通过 CANFD 挂载到主机的下位设备，包括传感器和执行器。
3. `module_uid`：模组出厂永久唯一标识，用于在自动编号、热插拔、心跳和重编号过程中稳定识别物理设备。
4. `逻辑号`：主机分配给某个模组的可读编号，首期建议按 `module_type + slot` 共同组成寻址键。
5. `preferred_slot`：模组当前期望使用的逻辑号，可由主机分配、历史记忆或模组按钮调整。
6. `assigned_slot`：主机当前实际分配给模组的逻辑号。
7. `inventory_revision`：主机当前设备清单和地址映射的版本号，地址变化时必须递增。
8. `能力`：模组暴露出来的标准化功能点，例如 `read.temperature`、`write.on`、`execute.blink`。
9. `Device Inventory`：主机当前真实接入的模组清单及其能力集合。
10. `Capability Registry`：平台维护的标准能力定义库，描述每种模组支持哪些操作、参数类型和范围。
11. `Project Manifest`：项目运行依赖的硬件、协议版本、资源预算和风险约束说明。
12. `Task DSL`：用户需求的结构化任务描述，表达触发条件、读取、判断、执行和安全限制。
13. `Logic Script`：云端生成并最终下发给主机运行的一段受约束脚本，不是完整固件。
14. `Release Package`：真正下发给主机的发布包，包含逻辑脚本、manifest、签名和元数据。
15. `发布闸门`：版本下发前必须通过的一组自动检查步骤。

# 系统整体架构

## 1. 架构分层

系统建议按四层设计：

1. 用户交互层：按键、屏幕、语音输入、语音播报、小程序和网页后台。
2. 云端编排层：项目管理、设备清单、对话记录、版本管理、日志查看。
3. 受约束生成层：方案 AI、DSL 生成器、静态校验器、代码生成器、沙盒测试器、发布服务。
4. 设备执行层：主机运行时、CANFD 总线、各类模组固件。

## 2. 各组件职责

### 2.1 主机

主机不只是“跑脚本的盒子”，还应承担以下职责：

1. 网络接入与账号绑定。
2. 与云端进行身份认证。
3. 模组扫描和能力发现。
4. 本地缓存设备清单。
5. 录音、屏幕显示、按键交互、语音播报。
6. 逻辑脚本沙盒运行和进程隔离。
7. 日志收集、异常检测和自动回滚。
8. 签名包校验和版本切换。
9. 运行时调用校验和保护。

### 2.2 云端 API 服务

云端 API 服务负责：

1. 用户账号和设备绑定。
2. 项目创建与版本管理。
3. 设备清单同步。
4. 对话记录和方案记录。
5. 发布状态追踪。
6. 社区工程管理。

### 2.3 方案 AI

方案 AI 只负责：

1. 理解自然语言需求。
2. 判断需要哪些模组。
3. 形成结构化方案。
4. 识别模糊点并给出确认项。
5. 生成 Task DSL。

方案 AI 不直接输出最终下发包。

### 2.4 规则引擎与校验器

规则引擎负责：

1. 校验设备是否真实存在。
2. 校验能力是否可用。
3. 校验风险动作是否允许。
4. 校验参数是否越界。
5. 校验资源预算是否超标。
6. 形成可读的错误报告。

### 2.5 代码生成器

代码生成器负责：

1. 将 Task DSL 转成 Logic Script。
2. 使用统一模板填充项目结构。
3. 保证输出脚本只调用允许 API。
4. 生成可被静态分析的受限 Lua 代码。

### 2.6 发布服务

发布服务负责：

1. 触发静态校验。
2. 触发沙盒仿真。
3. 生成版本号和签名。
4. 保留候选版本和回滚版本。
5. 向主机下发可部署包。

## 3. 首期是否需要 MCP

首期不需要把整个系统做成 MCP 服务器。

原因：

1. 这个项目的关键问题不是“模型如何调用工具”，而是“模型能否被严格限制在能力白名单里”。
2. 首期只要云端能把设备清单、能力表和规则注入 prompt，并能对生成结果做静态校验，就已经足够。
3. MCP 适合作为后续扩展，让模型调用 `get_device_inventory`、`validate_program` 等服务，但不是 MVP 必要条件。

首期建议：

1. 云端内部保留普通服务接口。
2. 模型只接触 prompt 和受限输出格式。
3. 不开放“直接下发到设备”的模型工具。

## 4. 整体链路

建议采用以下主链路：

`用户需求 -> 语音转文本 -> AI 生成结构化方案 -> 用户确认 -> AI 生成 Task DSL -> 规则引擎校验 -> 代码生成器输出 Logic Script -> 静态校验 -> 沙盒验证 -> 签名发布 -> 主机部署 -> 运行监控与回滚`

这个链路中最关键的是两点：

1. AI 不能直接从自然语言跳到最终可运行代码。
2. 所有可执行结果都必须落在“设备实际能力集合”之内。

# 设备初始化与能力发现

## 1. 首次初始化流程

用户首次使用时，建议执行以下步骤：

1. 主机联网并完成账号注册。
2. 云端为主机分配唯一设备 ID、设备证书或密钥、项目空间和默认 MQTT 主题。
3. 主机扫描当前接入的模组。
4. 每个模组通过 CANFD 向主机主动上报基础信息，其中必须包含 `module_uid`。
5. 主机对每个模组发起能力发现，并读取其历史 `preferred_slot` 或未分配状态。
6. 主机为每个模组分配 `assigned_slot`、心跳周期和租约超时时间。
7. 主机建立本地设备清单和地址映射表。
8. 主机把设备清单同步到云端。

初始化完成后，系统中应该存在一份真实的设备清单，而不是仅靠用户口述。

## 2. 热插拔与重新发现

模组不是只在开机时发现一次，还需要支持热插拔：

1. 新模组接入后主动 `Hello`，其中包含 `module_uid`、`preferred_slot`、`assigned_slot` 和地址状态。
2. 主机收到后触发 `Discover` 和 `GetCapabilities`。
3. 主机根据当前在线同类模组集合完成自动编号或重编号。
4. 主机更新本地缓存、地址映射和云端镜像，并递增 `inventory_revision`。
5. 如果某项目正在运行且依赖该逻辑号，主机要判断是否需要重建运行图或重启脚本。
6. 模组拔出时不要求主动通知，但主机应通过心跳租约超时判断离线。

## 3. 版本不匹配处理

若发现模组协议版本或固件版本不兼容，主机不能静默忽略，应明确记录：

1. 当前模组种类。
2. 当前槽位。
3. 发现到的协议版本。
4. 最低要求版本。
5. 是否允许只读兼容。
6. 是否需要 OTA。

## 4. 自动编号与心跳管理

为了减少小白用户接线和拨码的负担，首期建议取消人工拨码，改为自动分配逻辑号。

### 4.1 设计原则

1. 每个模组必须有出厂永久唯一的 `module_uid`。
2. 主机负责逻辑号分配，不由用户通过拨码硬编码。
3. 逻辑号建议按 `module_type + assigned_slot` 组成寻址键。
4. 新模组插入后若尚未分号，使用特殊值 `0xFF` 表示 `unassigned`。
5. 主机通过心跳租约判断模组是否仍在线。
6. 同型号模组可通过板载两个按钮调整目标逻辑号，主机收到请求后自动重分发。

### 4.2 为什么当前协议还不够

以当前文档原始版本来看，虽然已经有 `Hello`、`Heartbeat`、`Discover` 和 `Configure`，但还存在以下缺口：

1. 没有 `module_uid`，无法稳定区分多个同型号模组。
2. 没有 `unassigned` 状态，无法表达“刚插上但尚未分号”的模组。
3. 没有地址分配状态机，无法定义申请、分配、确认和持久化流程。
4. 没有心跳租约规则，无法确定丢几次算离线。
5. 没有按钮改号事件定义，无法支持从 `5` 调整到 `3` 这种需求。
6. 没有冲突重分发策略，多个模组争抢相同逻辑号时行为不确定。
7. 没有地址映射版本号，编号变化后无法判断脚本绑定是否已过期。

### 4.3 逻辑号作用域

为了降低复杂度，首期建议把逻辑号定义为“同类型模组内唯一”，而不是全局唯一。

也就是说：

1. `relay + slot 1` 和 `soil_moisture + slot 1` 可以同时存在。
2. `send_cmd("relay", 1, "write.on", 1)` 与 `send_cmd("soil_moisture", 1, "read.percent", nil)` 不冲突。
3. 自动重分发只在相同 `module_type` 内进行，不影响其他类型模组。

### 4.4 心跳建议

协议应支持可配置心跳周期和离线租约：

1. 推荐默认 `heartbeat_interval_sec = 10`。
2. 推荐默认 `lease_timeout_sec = 30`。
3. 若为了省流量或降低总线占用，也可以配置 `30 秒一次心跳`，但建议至少 `90 秒` 才判定离线。
4. 主机应允许不同模组类型配置不同心跳周期。

### 4.5 当前建议

若以用户体验优先，建议默认：

1. 10 秒一次心跳。
2. 连续 3 次未收到则离线。
3. 地址变化时递增 `inventory_revision`。

## 5. 地址重分发规则

### 5.1 新模组插入

当一个未分配模组接入时：

1. 模组发送 `Hello`，其中 `assigned_slot = 0xFF`。
2. 若存在历史绑定记录，主机优先尝试恢复上次 `assigned_slot`。
3. 若历史号位已被占用，则按重分发规则重新安排。
4. 若没有历史记录，则优先分配最小可用号位。

### 5.2 已分配模组重新插入

当一个已经分过号的模组重新接入时：

1. 模组在 `Hello` 中带上 `module_uid` 和上次 `assigned_slot`。
2. 主机若确认该号位仍属于该 `module_uid`，直接恢复。
3. 若该号位已被其他在线模组占用，则进入冲突处理流程。

### 5.3 按钮改号

若模组上有两个按钮用于改号，建议定义为：

1. `slot_down`：将 `preferred_slot` 减 1。
2. `slot_up`：将 `preferred_slot` 加 1。
3. 模组本地只改变 `preferred_slot`，不自行宣布分配成功。
4. 模组必须立即向主机发送“改号请求事件”。
5. 最终号位以主机下发的 `assigned_slot` 为准。

### 5.4 重分发算法

为了满足“从 5 号调整为 3 号，上报后自动重分发”的需求，建议采用稳定重排算法。

对于同类型模组：

1. 若目标号位空闲，则直接分配。
2. 若目标号位被占用，且请求模组已经有旧号位，则使用“区间平移”。
3. 例如模组 A 从 `5 -> 3`，则原先 `3`、`4` 的模组顺次右移为 `4`、`5`，A 占用 `3`。
4. 例如模组 A 从 `3 -> 5`，则原先 `4`、`5` 的模组顺次左移为 `3`、`4`，A 占用 `5`。
5. 若是新插入未分配模组请求一个已占用号位，则主机从目标号位开始寻找最近空位，并对区间做稳定平移。
6. 若同类型号位已经满员，则拒绝分配并返回冲突错误。

### 5.5 重分发后必须做的事

每次地址重分发后，主机必须：

1. 更新本地 `module_uid -> assigned_slot` 映射。
2. 递增 `inventory_revision`。
3. 记录地址变化日志。
4. 将最新映射同步到云端。
5. 对受影响的运行项目做热重载检查或重启。

# 项目开发流程

## 1. 创建任务

用户按下“新项目”按键后：

1. 主机在本地创建项目上下文。
2. 云端创建项目条目。
3. 当前设备清单、固件版本、协议版本、地址映射版本和资源预算一起写入项目 manifest。
4. 为该项目生成唯一版本命名空间。

## 2. 需求输入

1. 用户按下“对话”按键开始录音。
2. 结束录音后，音频流上传至服务器并转换为文本。
3. 系统将文本需求与当前项目上下文、设备清单一起提交给方案 AI。

## 3. 方案生成与反馈

云端 AI 输出的第一层结果不是代码，而是“结构化方案”，至少包含：

1. 项目目标。
2. 需要使用的模组。
3. 每个模组对应的能力调用。
4. 执行流程。
5. 用户确认点。
6. 风险提示。

主机通过屏幕和语音反馈给用户，用户可以继续追加需求或修改约束。

## 4. 方案锁定

在进入代码生成前，系统需要做一次自动检查：

1. 所需模组是否实际存在。
2. 模组固件版本是否满足要求。
3. 是否超出主机允许的同时接入数量。
4. 是否超出供电或电流预算。
5. 是否包含危险动作或受限操作。

如果失败，则回到方案迭代，不进入代码生成。

## 5. 受约束生成

当方案锁定后，系统进入正式生成流程：

1. AI 将自然语言方案转为 Task DSL。
2. 规则引擎检查 DSL 是否只使用已注册能力。
3. 代码生成器依据模板、模组驱动封装和 DSL 生成 Logic Script。
4. 生成结果进入静态分析和沙盒测试。
5. 通过后生成签名包，供主机拉取。

## 6. 部署与运行

1. 主机拉取签名包。
2. 在 LuatOS 沙盒中启动新版本。
3. 沙盒先进行预热运行和健康检查。
4. 成功后切换为当前活动版本。
5. 若启动失败、心跳丢失或运行异常，则自动回滚到上一个稳定版本。

## 7. 云端管理及社区共享

### 7.1 个人后台

支持查看：

1. 历史项目对话记录。
2. 结构化方案。
3. Task DSL。
4. Logic Script。
5. 已发布版本。
6. 沙盒运行日志。
7. 当前设备清单、地址映射版本与固件版本。

### 7.2 远程下发

允许在后台将某个已验证的历史版本重新下发到指定主机，但必须满足：

1. 主机在线。
2. 目标主机的设备能力集与该项目要求匹配。
3. 版本包签名有效。
4. 目标主机具备回滚条件。

### 7.3 社区共享

社区共享建议分为“分享方案”和“分享可执行项目”两层：

1. 分享方案：分享需求、思路、模组组合和流程逻辑，不直接允许执行。
2. 分享可执行项目：必须带 manifest、协议版本、依赖模组清单、风险标签和签名信息。

克隆社区工程时，系统必须先做兼容性检查，不能直接盲目下发。

# 结构化中间层设计

## 1. 为什么必须有中间层

如果系统直接让 AI 从用户需求生成最终代码，会遇到以下问题：

1. 容易调用不存在的 API。
2. 容易假设用户接入了并不存在的模组。
3. 很难做自动校验和差错定位。
4. 很难做版本兼容与社区复用。

因此需要引入机器可校验的中间层。

## 2. 项目中建议保留的五类结构化对象

### 2.1 Device Inventory

记录当前主机接入了哪些模组，包括：

1. `module_uid`。
2. 模组种类。
3. `preferred_slot`。
4. `assigned_slot`。
5. 地址状态。
6. 模组固件版本。
7. 协议版本。
8. 支持能力列表。
9. 参数范围。
10. 资源占用。
11. 心跳周期和租约超时。

### 2.2 Capability Registry

记录平台对每一种模组的标准能力定义，包括：

1. 模组类型名称。
2. 模组类型编码。
3. 允许的能力 ID。
4. 每个能力的读写属性。
5. 参数类型和取值范围。
6. 风险等级。
7. 限速、冷却时间和持续时间上限。

### 2.3 Project Manifest

记录项目运行需要的约束，包括：

1. 项目版本。
2. 依赖模组清单。
3. 最低协议版本。
4. 最低模组固件版本。
5. 地址映射版本或清单哈希。
6. 资源预算。
7. 是否含受限动作。
8. 回滚策略。
9. 发布风险等级。

示例：

```yaml
project_id: proj_20260313_001
project_name: smart_watering
manifest_version: 1
min_host_runtime: 1.0.0
protocol_version: 1
inventory_revision: 42
requires:
  - type: relay
    slot: 1
    min_fw: 1.0.0
  - type: soil_moisture
    slot: 1
    min_fw: 1.0.0
budgets:
  max_parallel_actuators: 1
  max_loop_hz: 1
  max_script_mem_kb: 64
safety:
  risk_level: medium
  requires_user_confirm: true
  rollback_on_startup_error: true
```

### 2.4 Task DSL

用于描述任务逻辑，建议表达：

1. 触发条件。
2. 感知输入。
3. 判定规则。
4. 执行动作。
5. 周期任务。
6. 告警与异常策略。
7. 安全限制。

示例：

```yaml
project: smart_watering
triggers:
  - type: interval
    every_sec: 30
inputs:
  - id: soil
    from: soil_moisture.slot_01.read.percent
flow:
  - if:
      lt: [soil, 35]
    then:
      - action: relay.slot_01.write.on
        value: 1
      - delay_ms: 5000
      - action: relay.slot_01.write.on
        value: 0
safety:
  max_run_sec_per_hour: 120
  cooldown_sec: 30
```

### 2.5 Release Package

实际下发给主机的发布包需要包含：

1. 逻辑脚本文件。
2. manifest。
3. 目标协议版本。
4. 目标模组清单。
5. 版本号。
6. 签名。
7. 哈希值。
8. `inventory_revision` 或映射哈希。
9. 回滚指针。
10. 发布记录 ID。

# 云端生成系统设计

## 1. 输入给模型的内容

云端每次生成时，建议至少传入以下信息：

1. 当前自然语言需求文本。
2. 历史对话摘要。
3. Device Inventory。
4. Capability Registry 中与当前设备相关的部分。
5. 平台允许使用的脚本 API。
6. 风险动作清单。
7. 不允许输出的语法和库。
8. 目标输出格式要求。

## 2. 模型输出阶段

建议把模型输出拆成两个阶段，而不是一步到位：

### 阶段 A：结构化方案

输出：

1. 项目目标。
2. 所需模组。
3. 逻辑流程。
4. 需要用户确认的点。
5. 风险提示。

### 阶段 B：Task DSL

输出：

1. 触发器。
2. 输入。
3. 流程。
4. 动作。
5. 安全限制。

是否让模型直接输出 Logic Script，可以作为第三阶段，但不建议在 MVP 首期把脚本完全放给模型自由生成。更稳的方式是：平台由 DSL 生成脚本。

## 3. 云端代码的定位

如果后续确实需要让模型产出一段 Lua 代码，这段代码也应定义为：

1. 只是逻辑脚本，不是完整固件。
2. 只运行在平台提供的固定沙盒模板中。
3. 只能调用白名单 API。
4. 不能访问文件、网络、系统命令和原始外设接口。

## 4. 首期推荐的脚本模式

首期建议固定脚本外壳，只让生成器填充逻辑片段：

1. 平台提供 scheduler。
2. 平台提供日志函数。
3. 平台提供 `send_cmd`。
4. 平台提供 `delay_ms`。
5. 平台提供少量状态变量接口。

这样可以把 AI 的自由度限制在可分析范围内。

# Logic Script 与 send_cmd 设计

## 1. Logic Script 目标

Logic Script 是主机沙盒运行的一段受约束 Lua 脚本，其目标不是表达所有计算能力，而是表达：

1. 什么时候读。
2. 读哪个模组。
3. 何时判断。
4. 何时执行。
5. 执行多久。

## 2. 首期脚本允许的语法子集

为方便静态校验，首期建议只允许：

1. `local` 常量赋值。
2. `if / elseif / else`。
3. 平台提供的定时器注册。
4. 简单比较运算。
5. 简单算术运算。
6. 白名单函数调用。
7. `return`。

首期不允许：

1. 自定义模块 `require`。
2. 文件操作。
3. 网络访问。
4. 动态加载代码。
5. 协程。
6. 无上界 `while true` 循环。
7. 任意递归。
8. 直接访问底层 CANFD 原始发送接口。

## 3. 首期白名单 API

建议首期只暴露以下 API：

1. `send_cmd(module_type, slot, action, value)`
2. `delay_ms(ms)`
3. `log_info(msg)`
4. `log_warn(msg)`
5. `get_var(name)`
6. `set_var(name, value)`

后续如有需要，再考虑：

1. `subscribe_event(...)`
2. `cancel_subscription(...)`
3. `now_ms()`

## 4. send_cmd 设计目标

`send_cmd` 是平台对模组控制的唯一脚本入口。它的职责不是简单转发，而是：

1. 检查调用是否合法。
2. 将逻辑调用映射成 Capability Registry 中定义的能力。
3. 将合法调用编码为 CANFD 协议帧。
4. 跟踪响应、重试和错误码。
5. 执行运行时保护。

## 5. 地址域约定

为了兼容自动编号和按钮改号，脚本层应明确约定：

1. `send_cmd` 的寻址键是 `(module_type, slot)`。
2. 这个 `slot` 不是物理拨码号，而是主机当前分配的逻辑号。
3. 主机内部必须维护 `module_uid -> module_type -> assigned_slot` 映射。
4. 地址变化后，主机必须刷新映射并递增 `inventory_revision`。
5. 若运行中的脚本依赖旧映射，主机必须阻止继续执行或重启受影响项目。

## 6. send_cmd 推荐签名

首期建议使用固定签名：

```lua
send_cmd(module_type, slot, action, value)
```

字段说明：

1. `module_type`：字符串或枚举，如 `relay`、`soil_moisture`。
2. `slot`：整数逻辑号，作用域限定在对应 `module_type` 内。
3. `action`：字符串，建议格式为 `<verb>` 或 `<verb>.<capability>`。
4. `value`：本次动作附带的值，可为 `nil`、整数、浮点、布尔、字符串。

### 推荐动作格式

1. `read.value`
2. `read.temperature`
3. `write.on`
4. `write.color`
5. `execute.blink`
6. `subscribe.threshold`

## 7. send_cmd 示例

### 7.1 简单继电器控制

```lua
send_cmd("relay", 1, "write.on", 1)
delay_ms(5000)
send_cmd("relay", 1, "write.on", 0)
```

### 7.2 读取土壤湿度

```lua
local value = send_cmd("soil_moisture", 1, "read.percent", nil)
if value < 35 then
  log_info("soil is dry")
end
```

### 7.3 舵机转动

```lua
send_cmd("pwm_servo", 3, "write.angle", 90)
```

## 8. send_cmd 内部执行流程

主机内部收到 `send_cmd` 调用后，建议执行以下顺序：

1. 检查 `module_type` 是否是字面量且在白名单中。
2. 检查当前 `inventory_revision` 是否仍与脚本启动时一致，若不一致则进入地址重绑定流程。
3. 检查 `(module_type, slot)` 是否存在于当前 Device Inventory。
4. 检查该逻辑号映射到的实际 `module_uid` 是否在线。
5. 检查 `action` 是否在该模组 capability 白名单中。
6. 检查 `value` 类型是否匹配。
7. 检查 `value` 是否越界。
8. 检查调用频率、冷却时间、最大持续时间。
9. 检查该动作是否属于高风险动作。
10. 将调用编码为 CANFD 帧并发送。
11. 等待响应并处理错误码。

如果任一步失败，主机应拒绝执行并记录日志。

## 9. 地址变更后的脚本处理

自动编号和按钮改号引入了一个新问题：脚本中的逻辑号可能在运行期间失效。

因此主机必须至少选择以下一种策略：

1. 地址变化后立即暂停受影响项目并重新启动。
2. 地址变化后自动重新注入新的绑定表，再恢复执行。
3. 地址变化后仅保留不依赖受影响模组的任务，其余任务暂停。

首期最稳的策略建议是第 1 种：地址变更就重启受影响项目。

## 10. 运行时保护不能省略

即使云端已经做过静态校验，主机侧也必须再做一次运行时保护。

原因：

1. 静态校验只能看到代码文本，看不到全部运行时状态。
2. 设备可能在发布后发生热插拔。
3. 模组版本可能在运行前后发生变化。
4. 某些参数可能由运行时状态计算得出。

# 静态校验设计

## 1. 静态校验的定义

静态校验是指：在脚本还没有真正运行之前，只根据“脚本文本、DSL、manifest、Device Inventory、Capability Registry”判断该版本是否符合平台规则。

它不依赖真实设备动作结果，也不依赖用户手工测试。

## 2. 静态校验的目标

静态校验要解决的问题不是“证明程序绝对正确”，而是优先挡住这些高概率问题：

1. 调用不存在的 API。
2. 访问不存在的模组。
3. 访问错误槽位。
4. 调用不支持的能力。
5. 传入越界参数。
6. 生成超高频或危险控制逻辑。
7. 绕过平台安全层。

## 3. 静态校验输入

每次校验应至少拿到：

1. Logic Script。
2. 生成该脚本的 Task DSL。
3. Project Manifest。
4. Device Inventory。
5. Capability Registry。
6. 当前 `inventory_revision`。
7. 平台安全规则版本号。

## 4. 静态校验步骤

### 4.1 脚本可解析性校验

1. 代码必须语法正确。
2. 代码必须能被 Lua AST 解析器解析。
3. 若存在语法错误，直接拒绝进入后续步骤。

### 4.2 语法子集校验

只允许首期定义的 Lua 子集：

1. `local`。
2. `if / elseif / else`。
3. 常量表达式。
4. 白名单函数调用。

发现以下情况直接拒绝：

1. `require`。
2. `load`、`loadstring`。
3. 文件操作。
4. 网络操作。
5. 非白名单全局变量访问。
6. 自定义无限循环。

### 4.3 API 白名单校验

扫描所有函数调用，只允许：

1. `send_cmd`。
2. `delay_ms`。
3. `log_info`。
4. `log_warn`。
5. `get_var`。
6. `set_var`。

出现其他函数调用则拒绝。

### 4.4 send_cmd 调用提取

从 AST 中提取所有 `send_cmd(module_type, slot, action, value)` 调用，并生成调用表。

每条调用记录至少包含：

1. 出现位置。
2. 模组类型。
3. 槽位号。
4. 动作名。
5. 值字面量。
6. 所处条件块。

### 4.5 send_cmd 参数校验

首期强烈建议以下三项必须是静态字面量：

1. `module_type` 必须是字符串字面量。
2. `slot` 必须是整数常量。
3. `action` 必须是字符串字面量。

`value` 允许：

1. `nil`
2. 数字字面量
3. 布尔字面量
4. 字符串字面量
5. 来自白名单变量的简单表达式

如果 `module_type`、`slot`、`action` 由复杂表达式动态计算，首期直接拒绝。

### 4.6 设备存在性校验

对每条 `send_cmd`：

1. `(module_type, slot)` 是否存在。
2. 该逻辑号是否真的映射到一个在线 `module_uid`。
3. 该模组是否在线。

不满足则拒绝发布。

### 4.6.1 地址稳定性校验

自动编号会导致地址映射变化，因此静态校验还必须增加一项：

1. 本次脚本校验所依据的 `inventory_revision` 必须写入发布包。
2. 若从静态校验到真正发布之间 `inventory_revision` 发生变化，则必须重新校验。
3. 若主机拉取版本时发现本地 `inventory_revision` 与发布包不一致，则必须拒绝直接启用，改为重新绑定或重新生成。

### 4.7 能力合法性校验

对每条 `send_cmd`：

1. `action` 是否存在于该模组 capability 白名单。
2. 对应能力是否允许当前风险级别项目调用。
3. 是否需要用户确认。

不满足则拒绝发布。

### 4.8 参数范围校验

对每条 `send_cmd`：

1. 值类型是否匹配。
2. 值是否在最小值和最大值范围内。
3. 枚举值是否在允许集合内。
4. 是否违反模组冷却时间。
5. 是否违反单次持续时间上限。

### 4.9 行为风险校验

校验逻辑脚本是否存在明显危险模式，例如：

1. 高频开启继电器。
2. 长时间持续驱动舵机或电机。
3. 同时驱动多个高功耗执行器。
4. 在没有明确停止动作时持续执行。
5. 没有超时保护的危险动作。

### 4.10 项目约束校验

将脚本行为与 manifest 对比：

1. 是否超出最大并行执行器数量。
2. 是否超出最大执行频率。
3. 是否超出脚本内存预算。
4. 是否违反项目风险等级限制。

## 5. 静态校验输出

校验器最终应输出机器可读结果和人类可读报告。

建议结构：

```yaml
status: failed
errors:
  - code: SLOT_NOT_FOUND
    message: slot 4 does not exist for relay
    line: 12
  - code: ACTION_NOT_ALLOWED
    message: write.speed is not supported by relay
warnings:
  - code: HIGH_FREQUENCY_CALL
    message: relay toggles more than 10 times per minute
summary:
  send_cmd_calls: 4
  risk_level: high
```

## 6. 静态校验不做什么

静态校验不负责：

1. 证明业务逻辑一定满足用户真实意图。
2. 证明所有实时条件下都不会出错。
3. 代替运行时保护。

所以静态校验不是唯一安全措施，而是第一道门。

# 发布闸门设计

## 1. 发布闸门的定义

发布闸门不是一句抽象口号，而是一个明确的自动化流程：

只有满足所有规则的版本，才允许被标记为“可下发”。

它的作用是把“AI 刚生成的草稿”与“可以真正推送到主机的版本”区分开。

## 2. 版本状态机

建议每个项目版本至少有以下状态：

1. `draft`：刚创建，只有需求和对话。
2. `planned`：结构化方案已生成。
3. `locked`：用户已确认方案。
4. `dsl_validated`：Task DSL 已通过规则校验。
5. `script_generated`：Logic Script 已生成。
6. `static_checked`：静态校验通过。
7. `simulated`：沙盒仿真通过。
8. `signed`：已生成签名包。
9. `deployable`：允许主机拉取。
10. `deployed`：已在主机部署。
11. `address_mismatch`：校验通过后地址映射发生变化，需重新验证。
12. `rolled_back`：已回滚到上一个稳定版本。

## 3. 每道闸门要检查什么

### 3.1 方案锁定闸门

进入条件：

1. 用户需求已明确。
2. 方案已生成。
3. 必需模组均存在。

输出结果：

1. 允许进入 DSL 生成。
2. 或返回用户补充信息。

### 3.2 DSL 校验闸门

检查：

1. DSL 结构是否合法。
2. DSL 只引用已存在模组。
3. DSL 只调用白名单能力。
4. DSL 是否含高风险动作。

### 3.3 Script 生成闸门

检查：

1. 脚本生成是否成功。
2. 生成结果是否只使用白名单 API。
3. 是否成功保留源 DSL 到脚本的映射关系。

### 3.4 静态校验闸门

检查：

1. 语法正确。
2. `send_cmd` 参数合法。
3. 项目约束不冲突。
4. 风险动作符合限制。

### 3.5 沙盒仿真闸门

检查：

1. 脚本能否正常启动。
2. 基础调用能否完成。
3. 是否会异常退出。
4. 是否触发 watchdog。
5. 是否能得到预期响应或错误处理。

### 3.6 签名闸门

检查：

1. 版本包内容完整。
2. manifest 和脚本哈希一致。
3. manifest 中的 `inventory_revision` 与当前校验结果一致。
4. 签名成功。
5. 回滚包可用。

## 4. 发布闸门失败后的处理

不同闸门失败时，系统应进入不同处理分支：

1. DSL 校验失败：回到方案或 DSL 生成阶段。
2. 静态校验失败：返回生成器重新修正脚本。
3. 地址映射变化：进入 `address_mismatch`，要求重新校验或重新生成。
4. 沙盒仿真失败：阻止签名发布，并记录运行日志。
5. 签名失败：不允许进入 `deployable`。
6. 主机部署失败：自动触发回滚。

## 5. 为什么发布闸门很重要

没有发布闸门，系统就会出现一种危险状态：

1. AI 生成的第一稿直接进入设备。
2. 设备侧承担全部错误后果。
3. 用户难以定位是模型理解错、脚本错、协议错还是硬件错。

有发布闸门后：

1. 错误能在云端尽量被拦住。
2. 版本可追溯。
3. 可以只把“通过校验的版本”下发到主机。

# Logic Script 运行时保护

## 1. 运行时保护的必要性

即使脚本已经通过静态校验，主机仍必须做运行时保护，因为运行时环境会变化：

1. 模组可能被热插拔。
2. 模组可能离线。
3. 总线可能拥塞。
4. 用户可能强制停止。
5. 某些值可能由运行时条件计算得出。

## 2. send_cmd 运行时保护项

主机在每次 `send_cmd` 执行时都建议校验：

1. 当前运行版本是否仍有效。
2. 模组是否在线。
3. `inventory_revision` 是否仍一致。
4. capability 是否仍可用。
5. 动作是否被限流。
6. 动作是否处于冷却时间内。
7. 是否触发紧急停止状态。
8. 本次动作是否会超出总功耗预算。

## 3. Watchdog 与超时

建议主机保留：

1. 脚本级 watchdog。
2. 单次 CAN 请求超时。
3. 连续错误计数器。
4. 启动阶段健康检查超时。

达到阈值时：

1. 停止 Candidate 版本。
2. 上报错误日志。
3. 回滚到 Stable 版本。

## 4. 双槽运行策略

建议主机保留两个运行槽：

1. `Stable`：当前稳定版本。
2. `Candidate`：新版本。

新版本部署时：

1. 先写入 Candidate。
2. Candidate 启动预热。
3. 若健康检查通过，则切换为 Stable。
4. 若失败，则删除 Candidate 并恢复 Stable。

## 5. 硬件级保护

建议硬件层面保留以下保护：

1. 主机物理复位键。
2. 主机紧急停止键。
3. 高风险执行器单独供电或单独使能。
4. 继电器、电机、泵类模组默认上电关闭。
5. 关键动作需要限时运行和超时断开。

# 私有 CANFD 协议 v1

## 1. 设计目标

该协议的目的不是“把所有逻辑塞进一条数据帧”，而是建立一套统一、可扩展、可版本化的设备控制语言。

协议 v1 要解决的问题：

1. 设备发现。
2. 能力查询。
3. 数据读取。
4. 动作写入。
5. 事件上报。
6. 错误反馈。
7. 分片传输。
8. OTA 升级。

## 1.1 自动编号结论

现有协议的基础帧结构可以承载自动编号、心跳保活和按钮改号需求，不需要推翻重来。

但要真正满足需求，必须在现有协议语义上补齐以下内容：

1. `module_uid`。
2. `assigned_slot = 0xFF` 的未分配状态。
3. 地址分配状态机。
4. 心跳租约规则。
5. 改号请求事件。
6. 冲突重分发算法。
7. `inventory_revision` 和地址变化后的发布保护。

## 2. 基础帧结构

建议应用层统一使用 64 Byte 数据区，定义如下：

| 字节偏移 | 字段名称 | 长度 | 说明 |
| -------- | -------- | ---- | ---- |
| Byte 0 | 协议版本 | 1 Byte | 当前定义为 `0x01` |
| Byte 1 | 设备种类 | 1 Byte | `0x00` 为主机，其余为模组种类 |
| Byte 2 | 槽位号 | 1 Byte | 逻辑槽位，建议 1 到 16；`0xFF` 表示未分配；作用域限定在 `module_type` 内 |
| Byte 3 | 帧类型 | 1 Byte | 请求、响应、事件、心跳、分片、OTA 等 |
| Byte 4 | 操作码 | 1 Byte | 读、写、发现、配置、执行、升级等 |
| Byte 5 | 子功能码 | 1 Byte | 具体能力点或动作编号 |
| Byte 6 | 序号 Seq | 1 Byte | 用于去重、重发和顺序控制 |
| Byte 7 | 标志位 Flags | 1 Byte | ACK、错误、最后分片、广播等标记 |
| Byte 8-9 | Payload 长度 | 2 Byte | 实际数据长度，最大 50 |
| Byte 10-11 | Request ID | 2 Byte | 请求与响应关联 ID |
| Byte 12 | 错误码 | 1 Byte | 响应失败时使用 |
| Byte 13 | 编码方式 | 1 Byte | Boolean、Int32、Float、String、Raw 等 |
| Byte 14-63 | Payload | 50 Byte | 具体数据区 |

说明：

1. 单帧最大有效负载为 50 Byte。
2. 大于 50 Byte 的能力描述、日志、OTA 数据块必须走分片传输。
3. 所有请求都应尽量有对应响应，便于主机重试和错误定位。

## 9. 地址状态定义

| 编码 | 状态 | 说明 |
| ---- | ---- | ---- |
| 0x00 | Unassigned | 尚未被主机分配逻辑号 |

## 10. 地址管理子功能定义

不必新增基础帧结构，但建议对现有 `Configure`、`Event`、`Heartbeat` 补充子功能。

### 10.1 Configure 子功能

| 子功能码 | 名称 | 说明 |
| -------- | ---- | ---- |
| 0x01 | AssignSlot | 主机下发 `assigned_slot` |
| 0x02 | SetHeartbeat | 主机下发心跳周期和租约 |
| 0x03 | ClearSlot | 清空当前逻辑号，回到 `unassigned` |
| 0x04 | ForceReaddress | 要求模组重新进入地址分配流程 |

### 10.2 Event 子功能

| 子功能码 | 名称 | 说明 |
| -------- | ---- | ---- |
| 0x01 | SlotChangeRequest | 模组请求改号 |
| 0x02 | ButtonPressed | 模组上报加号或减号按钮事件 |
| 0x03 | AddressConflict | 模组检测到地址冲突 |

### 10.3 Heartbeat 子功能

| 子功能码 | 名称 | 说明 |
| -------- | ---- | ---- |
| 0x01 | Alive | 正常在线心跳 |
| 0x02 | LeaseRenew | 请求续租 |
| 0x03 | LeaseAck | 主机确认续租 |

## 11. 自动编号核心字段

建议以下字段作为地址管理最小必备信息：

1. `module_uid`
2. `module_type`
3. `preferred_slot`
4. `assigned_slot`
5. `address_state`
6. `heartbeat_interval_sec`
7. `lease_timeout_sec`
8. `inventory_revision`

## 12. 自动编号与重编号流程

### 12.1 首次插入

1. 模组发送 `Hello`，其中 `assigned_slot = 0xFF`。
2. 主机读取 `module_uid` 和 `preferred_slot`。
3. 主机根据当前同类型在线模组集合分配逻辑号。
4. 主机通过 `Configure.AssignSlot` 下发最终 `assigned_slot`。
5. 模组返回 ACK，并进入 `Assigned` 状态。
6. 主机下发 `Configure.SetHeartbeat`。

### 12.2 已知模组重连

1. 模组发送 `Hello`，带上历史 `assigned_slot`。
2. 主机若确认该号位仍可用，则直接恢复。
3. 若该号位不可用，则进入重分发。
4. 分配结束后主机递增 `inventory_revision`。

### 12.3 按钮改号

1. 用户按下 `slot_up` 或 `slot_down`。
2. 模组修改本地 `preferred_slot`。
3. 模组通过 `Event.SlotChangeRequest` 上报请求。
4. 主机运行重分发算法。
5. 主机向所有受影响模组重新下发 `assigned_slot`。
6. 主机更新 `inventory_revision`，并重启受影响项目。

## 14. 能力发现流程

建议标准流程如下：

1. 模组上电后发送 `Hello`。
2. 主机收到后发送 `Discover`。
3. 模组返回基础信息：`module_uid`、种类、`preferred_slot`、`assigned_slot`、协议版本、固件版本、制造信息。
4. 主机发送 `GetCapabilities`。
5. 模组返回能力清单，必要时走分片帧。
6. 主机将能力清单写入 Device Inventory。

能力清单至少应包含：

1. 能力 ID。
2. 能力名称。
3. 读写属性。
4. 数据类型。
5. 取值范围。
6. 单位。
7. 默认值。
8. 是否支持事件订阅。
9. 风险等级。

## 16. 心跳与离线判定

心跳机制建议采用“租约”思想，而不只是简单定时包。

### 16.1 推荐策略

1. 主机下发 `heartbeat_interval_sec`。
2. 模组按该周期发送 `Heartbeat.Alive`。
3. 主机在 `lease_timeout_sec` 内未收到心跳，则标记离线。
4. 离线后释放该模组占用的逻辑号，并递增 `inventory_revision`。

### 16.2 关于 30 秒一次心跳

`30 秒一次心跳` 是可以支持的，但需要明确体验代价：

1. 如果 30 秒发一次且丢 2 次判离线，则最快 60 秒发现离线。
2. 如果 30 秒发一次且丢 3 次判离线，则最快 90 秒发现离线。
3. 若想让拔插反馈更快，推荐 10 秒心跳、30 秒离线判定。

## 17. 事件与订阅

对于需要主动上报的模组，不建议让 AI 自己拼接比较逻辑，而应支持统一订阅模式：

1. 主机通过 `Subscribe` 注册阈值或触发条件。
2. 模组本地判断是否触发。
3. 触发后通过 `Event` 主动上报。

这样可以减少主机轮询压力，并让高实时性逻辑在模组侧闭环。

## 18. 地址变化对项目运行的影响

自动编号让系统更小白，但也带来一个工程问题：

1. 项目脚本是按 `(module_type, slot)` 调用模组的。
2. 如果用户在运行中把同类型模组从 `5` 改成 `3`，脚本中的旧映射就可能失效。

因此必须规定：

1. 每次地址变化都必须递增 `inventory_revision`。
2. 发布包必须记住校验时的 `inventory_revision`。
3. 运行中若发现地址版本变化，主机必须重启或重绑受影响项目。

## 19. OTA 基本流程

建议最小流程如下：

1. 主机发送 `OtaBegin`，声明目标版本和总大小。
2. 模组校验可升级条件并返回响应。
3. 主机分片发送 `OtaChunk`。
4. 模组逐片校验并回 ACK。
5. 主机发送 `OtaCommit`。
6. 模组写入成功后重启并重新发送 `Hello`。

OTA 失败时必须保留旧版本，不得写坏后无法恢复。

# 硬件设计建议

## 1. 主机职责

主机应承担以下职责：

1. 网络接入与账号绑定。
2. 与云端进行安全认证。
3. 模组扫描和能力发现。
4. 本地缓存设备清单。
5. 录音、屏幕显示、按键交互、语音播报。
6. 沙盒运行和进程隔离。
7. 日志收集、异常检测和自动回滚。
8. 签名包校验和版本切换。

## 2. 下位模组职责

每个模组应至少具备以下能力：

1. 上电自报身份。
2. 上报协议版本和固件版本。
3. 上报能力列表。
4. 响应主机查询。
5. 支持统一错误码。
6. 支持配置下发。
7. 支持心跳或状态上报。

对于复杂模组，还应支持：

1. 分片传输。
2. 参数校验。
3. 进度上报。
4. OTA 升级。

## 3. 主机资源建议

文档层面建议后续补齐这些指标：

1. 可用 RAM 预算。
2. 可用 Flash 预算。
3. 脚本运行内存上限。
4. 版本包最大尺寸。
5. 同时在线模组数量上限。
6. 总线带宽上限。

# 后台生成程序限制

这一节是系统能否落地的关键约束。

## 1. 总原则

1. AI 不允许直接输出“最终可运行代码并立即执行”。
2. AI 第一阶段只能输出结构化方案和 Task DSL。
3. 最终脚本必须由平台自己的代码生成器基于模板和能力映射生成，或由模型在严格子集约束下输出并再次被校验。

## 2. AI 允许做的事情

1. 理解自然语言需求。
2. 把需求拆解为感知、判断、执行三类步骤。
3. 生成结构化方案。
4. 生成 Task DSL。
5. 为用户生成解释文案、步骤说明和接线提示。
6. 在严格格式要求下生成受限 Logic Script。

## 3. AI 不允许做的事情

1. 调用未注册能力。
2. 访问不存在的模组槽位。
3. 生成绕过协议层的裸控制代码。
4. 修改主机安全策略。
5. 默认开启高风险执行器。
6. 在无确认的情况下执行持续性危险动作。
7. 访问网络、文件、系统命令。
8. 绕过 `send_cmd` 直接发送原始总线指令。

## 4. 代码生成前的自动校验

必须至少通过以下检查：

1. 设备存在性检查。
2. 协议版本兼容检查。
3. 模组固件版本兼容检查。
4. 参数范围检查。
5. 电流和供电预算检查。
6. 任务冲突检查。
7. 禁止动作检查。
8. 资源占用检查。

## 5. 代码生成后的自动校验

必须至少通过以下检查：

1. DSL 到脚本的映射完整性检查。
2. 语法检查。
3. AST 静态扫描。
4. `send_cmd` 参数提取与校验。
5. 模拟输入测试。
6. 沙盒运行测试。
7. 心跳和异常退出测试。

## 6. 失败处理策略

建议所有失败都返回结构化错误，而不是简单返回“生成失败”：

1. `device_missing`
2. `capability_not_allowed`
3. `param_out_of_range`
4. `unsafe_action_detected`
5. `script_parse_failed`
6. `sandbox_start_failed`

## 7. 发布闸门

只有在以下条件全部满足时才允许下发：

1. 结构化方案已锁定。
2. 所需硬件已确认实际存在。
3. 自动校验全部通过。
4. 版本包已签名。
5. 回滚包可用。

# 日志与可观测性设计

## 1. 为什么需要日志

这个系统的问题来源很多，包括：

1. 用户需求表达不清。
2. AI 方案误解。
3. DSL 错误。
4. 脚本错误。
5. 协议错误。
6. 模组固件错误。
7. 总线异常。

所以必须把日志设计为一等能力。

## 2. 建议的日志层级

1. `INFO`
2. `WARN`
3. `ERROR`
4. `SAFETY`

## 3. 建议记录的关键事件

1. 设备注册。
2. 模组接入和离线。
3. 方案锁定。
4. DSL 校验失败。
5. Script 静态校验失败。
6. 发布包签名成功。
7. Candidate 启动成功或失败。
8. 回滚发生。
9. 高风险动作被拒绝。
10. 自动编号成功。
11. 地址冲突。
12. 心跳超时离线。

# MVP 实施建议

## 1. 一期目标

一期目标不是覆盖大量模组，而是先跑通以下闭环：

`设备发现 -> 方案生成 -> Task DSL 校验 -> 脚本生成 -> 静态校验 -> 沙盒运行 -> 回滚 -> 云端查看日志`

## 2. 一期建议保留的模组

建议先只做 10 个左右模组，覆盖输入、输出、环境感知和简单执行：

1. 按键开关模块。
2. 三色 LED 模块。
3. WS2812B 模块。
4. 有源蜂鸣器。
5. 温湿度传感器。
6. 光照强度传感器。
7. 超声波测距模块。
8. 土壤湿度模块。
9. 人体存在传感器。
10. PWM 舵机模块。

这批模组的价值是：

1. 足够支撑大部分入门项目。
2. 覆盖读、写、事件、周期任务几种典型交互。
3. 风险相对可控，便于先跑通平台能力。

## 3. 二期再引入的模组

以下模组建议在协议、沙盒、回滚和安全模型成熟后再接入：

1. 继电器。
2. 直流水泵模组。
3. 直流风扇模组。
4. 无刷电机。
5. 电磁铁。
6. 各类气体、酒精、水质、PH 等工业或半工业传感器。
7. 指纹、人脸、摄像头、离线语音等复杂智能模组。

原因是这些模组要么带来更高执行风险，要么协议复杂度更高，要么需要额外的权限与隐私设计。

如果业务一定要在首期加入继电器，建议把继电器视为“受限高风险模组”，单独增加：

1. 默认关闭。
2. 单次持续时间上限。
3. 最小冷却时间。
4. 用户确认。
5. 物理紧急断开能力。

## 4. 一期推荐的示范项目

1. 智能夜灯。
2. 超声波测距报警器。
3. 土壤湿度提醒器。
4. 光照自动提示器。
5. 人体靠近触发提示器。
6. 简单舵机联动展示项目。

## 5. 一期开发顺序建议

建议按以下顺序推进：

1. 主机运行时和双槽回滚。
2. CANFD 发现与能力上报。
3. Capability Registry 与 Device Inventory。
4. Task DSL。
5. 静态校验器。
6. 脚本生成器。
7. 沙盒仿真和发布闸门。
8. Web 后台和项目管理。

# 总结

这套方案的关键不是“让 AI 更聪明”，而是“让 AI 只能在正确的边界内工作”。

这套系统的最小可信闭环应当是：

1. 真实设备先上报能力。
2. AI 只能基于真实能力生成方案。
3. 脚本只能调用白名单 `send_cmd`。
4. 云端先做静态校验和发布闸门。
5. 主机再做运行时保护和回滚。

如果后续继续按这个方向推进，系统核心竞争力会变成：

1. 好用的自然语言入口。
2. 完整的设备能力抽象。
3. 可验证、可回滚的自动生成链路。
4. 可以持续扩展的模组生态。
